# Design & build architecture

How the cookbook turns plain markdown into an art-directed PDF and a reflowable
EPUB. For day-to-day usage see the [README](README.md); to add a recipe see
[CONTRIBUTING](CONTRIBUTING.md).

## One source, two renderers

Recipes and book configuration are the single source of truth. One Python step
parses them and renders two HTML files; each is handed to a renderer tuned for
its medium.

```text
recipes/*.md + content/introduction.md + book.yaml
        │  tools/build_html.py  (Jinja2)
        ├──► build/cookbook.html  ──► tools/make_pdf.py   (WeasyPrint) ──► build/cookbook.pdf
        └──► build/epub.html      ──► tools/make_epub.sh  (pandoc)     ──► build/cookbook.epub
```

- `build/cookbook.html` is the art-directed print layout (`assets/css/print.css`):
  one fixed-size page element per sheet, with furniture absolutely positioned.
- `build/epub.html` is semantic, flat markup (`assets/css/epub.css`): pandoc splits
  it into one chapter file per recipe for a clean reflowable read.

## Why these tools

- **WeasyPrint** renders the PDF in pure Python — no headless browser. The print
  CSS deliberately relies only on what WeasyPrint supports natively: CSS Paged
  Media (`@page`, fixed pages, page breaks), flexbox, transforms, and ordinary
  raster backgrounds. There are no running headers or page counters to polyfill.
- **pandoc** produces a standards-compliant EPUB; `epubcheck` validates it.
- **poppler** (`pdfinfo`, `pdftoppm`) inspects the PDF and rasterises the cover.

This keeps the toolchain to one language runtime (Python) plus two small
binaries (pandoc, poppler). Java is needed only to run `epubcheck`, and only in
CI — locally it is optional and `validate.py` falls back to a structural check.

## Layout strategy (print)

`print.css` treats every `.page` as exactly one printed sheet at the
`6.75in × 9.5in` trim, with `overflow: hidden` and `break-after: page`. All
furniture (titles, columns, footer, sidebar) is absolutely positioned inside the
page. Two patterns are worth noting because they keep the layout renderer-portable:

- **Pre-baked raster backgrounds.** Paper grain and the navy line-art patterns are
  baked to images (`tools/bake_assets.py`) rather than composed live with CSS blend
  modes or SVG backgrounds, which renderers handle inconsistently.
- **Rotated sidebar via `transform`.** The vertical recipe label on the outer edge
  is a normal horizontal line rotated 90°, not `writing-mode`.

A recipe with a non-empty `story` gets a full story page (optional `headshot`);
without one it gets a compact opener. Either way the method page follows.

## Design tokens

`book.yaml` holds the palette, fonts, and section order; `build_html.py` injects
the palette as CSS custom properties and the font faces, so the look is edited in
one place.

- Trim: `6.75in × 9.5in` portrait.
- Palette: navy `#16203a`, cream `#faefdb`, soft-ink text.
- Type: Playfair Display (display) + Bitter (body), both OFL, embedded in both outputs.

## Illustrations

Each recipe ships with an on-brand **SVG placeholder** generated by
`tools/gen_illustrations.py`, so every build is green with no network or API.

Real artwork is generated by hand in any image model and dropped in as
`assets/illustrations/recipes/<slug>.png`; the build prefers a raster sibling over
the placeholder automatically, with no front-matter change. To stay coherent, the
whole set shares one locked style:

> A hand-painted watercolor and ink cookbook illustration of *the dish* — loose
> expressive ink outlines, transparent watercolor washes, soft cast shadow beneath
> the dish, realistic but painterly, a single isolated subject on a fully
> transparent background, no backdrop, no text. Export at ~1500 px wide as a
> transparent PNG.

`tools/bake_assets.py` rasterises the decorative navy patterns and bakes the paper
grain that the PDF consumes; rerun it after editing those source SVGs.

## Validation

Content and artwork are original, so validation is structural, not pixel-based
(`tools/validate.py`):

- every `recipes/*.md` front matter conforms to `schema/recipe.schema.json` (JSON Schema);
- the PDF has the expected page count and trim;
- the EPUB passes `epubcheck`;
- a `build/contact-sheet.png` thumbnail grid is produced for eyeballing the look.
