# Design & build architecture

How `ladle` turns plain markdown into an art-directed PDF and a reflowable EPUB.
For day-to-day usage see the [README](README.md); to author a theme see
[docs/THEMING.md](docs/THEMING.md).

## One source, two renderers

Recipes and book configuration are the single source of truth. One Python step
parses them and renders two HTML files; each is handed to a renderer tuned for
its medium.

```text
a book (book.yaml + recipes/*.md + content/introduction.md) + a theme
        │  ladle build → build_html.render  (src/ladle/build_html.py, Jinja2)
        ├──► build/cookbook.html  ──► make_pdf.render   (WeasyPrint) ──► build/cookbook.pdf
        └──► build/epub.html      ──► make_epub.render  (pandoc)     ──► build/cookbook.epub
```

- `build/cookbook.html` is the art-directed print layout (the theme's
  `css/print.css`): one fixed-size page element per sheet, furniture absolutely
  positioned.
- `build/epub.html` is semantic, flat markup (the theme's `css/epub.css`):
  pandoc splits it into one chapter file per recipe for a clean reflowable read.

## Why these tools

- **WeasyPrint** renders the PDF in pure Python — no headless browser. The print
  CSS deliberately relies only on what WeasyPrint supports natively: CSS Paged
  Media (`@page`, fixed pages, page breaks), flexbox, transforms, and ordinary
  raster backgrounds. There are no running headers or page counters to polyfill.
- **pandoc** produces a standards-compliant EPUB; `epubcheck` validates it.
- **poppler** (`pdfinfo`, `pdftoppm`) inspects the PDF and rasterises the cover.

This keeps the toolchain to one language runtime (Python) plus two small
binaries (pandoc, poppler). Java is needed only to run `epubcheck`, and only in
CI — locally it is optional and `ladle validate` falls back to a structural check.

## Layout strategy (print)

`print.css` treats every `.page` as exactly one printed sheet at the
`6.75in × 9.5in` trim, with `overflow: hidden` and `break-after: page`. All
furniture (titles, columns, footer, sidebar) is absolutely positioned inside the
page. Two patterns are worth noting because they keep the layout renderer-portable:

- **Pre-baked raster backgrounds.** Paper grain and the navy line-art patterns ship
  as pre-rendered raster files in the theme (`illustrations/patterns/`) rather than
  composed live with CSS blend modes or SVG backgrounds, which renderers handle
  inconsistently.
- **Rotated sidebar via `transform`.** The vertical recipe label on the outer edge
  is a normal horizontal line rotated 90°, not `writing-mode`.

A recipe with a non-empty `story` gets a full story page (optional `headshot`);
without one it gets a compact opener. Either way the method page follows.

The front matter includes a **Contents** page: recipes grouped by section, with
page numbers resolved at render time via CSS `target-counter()` against each
recipe opener's `id` — no manual page numbering. The same headings drive the
**PDF outline** (viewer bookmarks): WeasyPrint emits one bookmark per heading, so
`print.css` sets `bookmark-level: none` on the `<h2>` section labels to keep the
outline to just Contents, Introduction, and the recipe titles. `make_pdf.py`
flips the document's `PageMode` to `/UseOutlines` so viewers open with the
bookmark panel showing.

## Design tokens & themes

The look lives in a **theme** (see [docs/THEMING.md](docs/THEMING.md)). A theme's
`theme.yaml` declares the default palette, fonts, and the font files to embed;
`build_html.py` injects the palette as CSS custom properties and the font faces.
A `book.yaml` overrides any token per key, so one theme backs many books and a
minimal book still renders. The tool hardcodes no fonts or colours.

A theme ships executable Jinja, so it is treated as untrusted **data, never
code**: `build_html.make_env` renders every theme template in a
`jinja2.sandbox.SandboxedEnvironment`, which blocks private-attribute access,
imports, and unsafe callables. A template that trips the sandbox surfaces as a
friendly `ConfigError` naming the file, not a traceback. This is the security
gate that lets community themes install by path without a review step.

Defaults for the built-in `default` theme:

- Trim: `6.75in × 9.5in` portrait.
- Palette: navy `#16203a`, cream `#faefdb`, soft-ink text.
- Type: Playfair Display (display) + Bitter (body), both OFL, embedded in both outputs.

## Illustrations

Recipe artwork is **author-provided content that lives with the book**, not
generated by the tool. A recipe's `illustration:` front matter points at a file
under `assets/illustrations/recipes/` (SVG or a raster); the build prefers a
raster sibling over the SVG when both exist, so real art drops in with no
front-matter change. To stay coherent, keep the whole set in one locked style:

> A hand-painted watercolor and ink cookbook illustration of *the dish* — loose
> expressive ink outlines, transparent watercolor washes, soft cast shadow beneath
> the dish, realistic but painterly, a single isolated subject on a fully
> transparent background, no backdrop, no text. Export at ~1500 px wide as a
> transparent PNG.

Decorative **theme** art (cover/endpaper/back patterns, paper grain) belongs to
the theme, not the book. It ships as pre-baked raster files under the theme's
`illustrations/patterns/` (rasterised from the theme's source SVGs and committed),
so the PDF build consumes ready-made backgrounds with no generation step.

## Validation

Content and artwork are original, so validation is structural, not pixel-based
(`ladle validate`, `src/ladle/validate.py`):

- every recipe's front matter conforms to `src/ladle/schema/recipe.schema.json` (JSON Schema);
- the PDF has the expected page count and trim;
- the EPUB passes `epubcheck`;
- a `build/contact-sheet.png` thumbnail grid is produced for eyeballing the look.
